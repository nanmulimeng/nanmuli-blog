# AI技术日报方案评审与改进规划

> 领主将军阁下提供的 v4.0 方案已阅，以下指出关键问题并给出改进方向。
> **原则：不另起炉灶，基于已有 WebCollector 能力构建。**

---

## 一、v4.0 方案的五大问题

### 问题1：与现有资产完全割裂（最严重）

WebCollector 已经具备的能力：
- Crawl4AI 无头浏览器爬取
- Bing/Sogou/Google/DDG 搜索引擎整合
- AI 关键词扩展（2-3个变体，语境判断优先）
- AI 关键词优化（消除歧义/补充术语）
- 三层去重（URL精确 + SimHash + 标题Jaccard）
- 来源可信度评分（官方95/社区80/博客60/农场20）
- 内容质量多维度评分（字数/结构/代码密度/广告占比）
- AI 内容整理（单页/多页综合整理，去重合并）

**但 v4.0 方案完全忽视了这些资产**，重新设计了一套独立的：
- 四层去重体系（URL→ID→SimHash→语义）——已有三层，何必再建？
- 质量过滤体系（广告/标题党/付费墙）——已有 quality.py
- 数据源管理体系（RSS/API/爬虫/实时流/Webhook）——已有搜索引擎统一入口

**结果**：方案变成一个与现有系统并行的"第二套采集系统"，维护成本翻倍。

### 问题2：数据源过度分散，维护不可行

v4.0 设计了 **七大板块 × 平均6个数据源 = 约42个数据源**：

| 板块 | 数据源数量 | 问题 |
|------|-----------|------|
| 新闻 | 6个 | Techmeme/TechCrunch/36氪/HN/Ars/The Information |
| 技术好文 | 6个 | HN/Medium/Dev.to/掘金/知乎/阮一峰 |
| 论文 | 5个 | arXiv(cs.SE+PL)/PWC/ACM/IEEE |
| 开源 | 5个 | GH Trending/HN/PH/Reddit/Gitee |
| AI速递 | 4个 | arXiv(cs.AI)/OpenAI Blog/Google Research/Twitter |
| 创新推荐 | 7个 | PH/MIT/Gartner/HN/BetaList/小众软件/Newsletter |
| 一句话锐评 | - | 需要人工编辑 |

**每个数据源都需要**：API Key申请、反爬策略、格式解析、失效监控、健康检查。

**现实**：42个数据源中，至少一半会在3个月内失效（API限制改版、反爬升级、网站关闭）。维护这套体系需要全职工程师。

### 问题3：实时热点监控是伪需求

v4.0 设计了复杂的实时热点监控：
- Twitter/X Stream API（监控大V推文、热门话题）
- HN Top 30 变化检测
- Reddit r/programming 热门帖子
- 触发条件：1小时内热度突增 > 300%

**问题**：
1. Twitter API 现在收费极高（Basic $100/月，Pro $5000/月），个人项目无法承担
2. "热度突增300%"如何量化？HN的points、Reddit的upvotes、Twitter的retweets量纲不同，无法统一计算
3. 即使检测到了热点，从"检测"到"爬取"到"整理"到"发布"，15分钟根本不现实（爬取+AI整理至少需要3-5分钟/篇）
4. **真正的高频热点，用户不会等你的日报**——Twitter/即刻/微信群已经传遍了

### 问题4：人工审核与自动化目标根本矛盾

v4.0 在多个环节插入"人工审核"：
- 突发新闻："人工审核(15分钟)"
- 新闻终审："人工确认 → 3-5条"
- 技术好文："人工确认 → 2-3篇"
- 论文："人工确认 → 1-2篇"
- 开源项目："人工复核 → 3个项目"

**如果目标是"自动化日报"，这些人工环节就是瓶颈。**

如果日报需要人工逐条确认，那和"编辑手动收集+整理"有什么区别？自动化的价值在于**减少人工介入**，而不是把人工审核包装成"AI辅助"。

### 问题5：四层去重和事件聚类过度设计

v4.0 设计了：
- Layer 1: URL精确匹配
- Layer 2: ID匹配（DOI/项目URL）
- Layer 3: SimHash（海明距离≤3）
- Layer 4: 语义相似度（TF-IDF余弦相似度≥0.85）
- 事件聚类：LDA主题模型 + K-Means

**已有三层去重足够覆盖99%的场景。**

Layer 4（TF-IDF语义相似度）和事件聚类（LDA+K-Means）的问题：
1. LDA需要预先设定主题数K，不同日期K不同，无法自动化
2. TF-IDF余弦相似度对短文本（标题+摘要）效果差
3. 计算成本高（每篇需要与历史所有文章计算相似度）
4. **SimHash+标题Jaccard已经能捕获99%的重复/转载**

## 二、改进方向：基于 WebCollector 的日报架构

### 2.1 核心思路转变

| 维度 | v4.0 方案 | 改进方案 |
|------|----------|---------|
| 采集方式 | 42个独立数据源 | **搜索引擎+AI扩展关键词** |
| 去重 | 四层+LDA聚类 | **复用已有三层去重** |
| 质量过滤 | 独立体系 | **复用已有 quality.py** |
| 内容整理 | AI摘要+人工审核 | **复用已有 AI organizeMultiple** |
| 板块获取 | 每个板块独立采集 | **每个板块=一组关键词搜索任务** |
| 输出 | 固定七大板块 | **灵活板块，按需生成** |

### 2.2 板块 → 关键词映射（简化版）

不再维护42个数据源，而是用**关键词搜索+AI扩展**获取各板块内容：

```
📰 今日技术新闻
  └─ 关键词: "科技新闻", "Tech news today"
  └─ 扩展词: AI生成 ["科技动态", "技术行业新闻", "硅谷新闻"]
  └─ 搜索引擎: Bing (time_range=day)
  └─ 时间过滤: day（只取当天）

📖 技术好文
  └─ 关键词: "技术博客", "programming tutorial"
  └─ 扩展词: AI生成 ["深入浅出", "最佳实践", "源码分析"]
  └─ 搜索引擎: Bing (time_range=week)
  └─ 时间过滤: week

📄 学术论文
  └─ 关键词: "arxiv", "latest papers"
  └─ 扩展词: AI生成 ["最新论文", "研究进展"]
  └─ 搜索引擎: Bing + site:arxiv.org (time_range=week)
  └─ 时间过滤: week

⭐ 热门开源
  └─ 关键词: "GitHub trending", "开源项目推荐"
  └─ 扩展词: AI生成 ["热门仓库", "新工具推荐"]
  └─ 搜索引擎: Bing (time_range=week)
  └─ 时间过滤: week

🤖 AI速递
  └─ 关键词: "AI新闻", "LLM更新", "OpenAI"
  └─ 扩展词: AI生成 ["大模型动态", "AI产品发布"]
  └─ 搜索引擎: Bing (time_range=day)
  └─ 时间过滤: day
```

**优势**：
- 不需要维护任何数据源的API/爬虫/反爬策略
- 搜索引擎天然去掉了失效问题（Bing/Google不会"失效"）
- AI扩展关键词自动适配语境（已有 `expandKeywords` 实现）
- 时间过滤确保时效性（已有 `time_range` 实现）

### 2.4 板块简化建议

v4.0 的七大板块可以简化为 **五大板块**，去除重叠度高的：

| v4.0 板块 | 建议 | 理由 |
|----------|------|------|
| 📰 新闻 | ✅ 保留 | 高频，独立价值 |
| 📖 好文 | ✅ 保留 | 深度，独立价值 |
| 📄 论文 | ✅ 保留 | 专业，独立价值 |
| ⭐ 开源 | ✅ 保留 | 实用，独立价值 |
| 🤖 AI速递 | ❌ 合并到新闻 | AI新闻本身就是新闻的子集，单独板块导致重复 |
| 💡 创新推荐 | ❌ 合并到好文 | "新颖创新"就是"技术好文"的一个子类，边界模糊 |
| 💬 一句话锐评 | ❌ 去除 | 需要人工编辑，与自动化目标矛盾 |

**简化后五大板块**：新闻 / 好文 / 论文 / 开源 / 延伸阅读

## 三、关键设计决策

### 决策1：不维护独立数据源，全部走搜索引擎

**理由**：
- WebCollector 的核心能力就是"搜索引擎+AI扩展+爬取+整理"
- 数据源越多，失效越快，维护成本指数增长
- 搜索引擎是"通用数据源"，永远不会失效

**成本对比**：
| 方案 | 维护成本/月 | 失效风险 |
|------|------------|---------|
| 42个独立数据源 | 2-3人天 | 高（每月2-5个失效） |
| 搜索引擎+关键词 | 0.5人天 | 极低 |

### 决策2：复用已有去重和质量体系，不另建

**理由**：
- `dedup.py` 的三层去重（URL+SimHash+标题）已经能处理99%的重复
- `quality.py` 的来源可信度+内容质量评分已经覆盖广告/标题党/内容农场
- 重建一套意味着两套逻辑不一致，bug翻倍

### 决策3：人工审核改为"置信度阈值"机制

**理由**：
- 人工审核是自动化日报的瓶颈
- 用已有质量评分体系替代人工判断：
  - `final_score >= 65` + `verdict == pass` → 自动入选（高置信）
  - `final_score 45-65` + `verdict == review` → 可选保留（中置信）
  - `final_score < 45` + `verdict == reject` → 自动丢弃（低置信）

### 决策4：日报是自动优化引擎的第一个用例

**理由**：
- 日报每天固定生成，是测试自动优化引擎的完美场景
- 每次生成后可以评估：各板块覆盖度、来源多样性、重复率
- 评估结果自动优化第二天的关键词策略
- 形成"生成→评估→优化→再生成"的闭环

| 内容去重（三层） | ✅ 已完成 |
| 质量评分+过滤 | ✅ 已完成 |
| AI关键词扩展/优化 | ✅ 已上线 |
| AI内容整理 | ✅ 已上线 |
| time_range 全链路传递 | ✅ 刚完成 |

### Phase 1：日报任务类型（1-2周）

**目标**：在 WebCollector 中新增 `DAILY_DIGEST` 任务类型

| 任务 | 说明 |
|------|------|
| 新增 CollectTaskType.DAILY_DIGEST | 日报类型枚举 |
| 新增板块配置 YAML | 定义各板块的关键词/时间范围/数量 |
| 修改 WebCollectorAsyncExecutor | 新增 DAILY_DIGEST 分支，串行执行各板块搜索 |
| 日报结果聚合 | 将各板块结果合并为一个 Markdown 输出 |

### Phase 2：定时调度（1周）

**目标**：使用 APScheduler 实现每日定时生成

| 任务 | 说明 |
|------|------|
| APScheduler 集成 | 每日8:00自动触发日报任务 |
| 周末处理 | 周五预采集/周一发布（或跳过周末） |
| 任务状态持久化 | 日报任务的状态追踪 |

### Phase 3：自动优化闭环（2-3周）

**目标**：日报生成后自动评估并优化下一次策略

| 任务 | 说明 |
|------|------|
| Coverage Evaluator | 评估各板块覆盖度（来源数量/质量/多样性） |
| 策略知识库 | 记录"XX关键词+XX引擎+XX时间范围"的效果 |
| 自动调整 | 覆盖度不足的板块自动调整关键词/时间范围 |

## 六、总结

v4.0 方案的核心问题是**在已有完备能力的基础上另起炉灶**。改进方案的原则是：

1. **不复建**：已有的 dedup/quality/AI整理/关键词扩展 全部复用
2. **不维护数据源**：全部走搜索引擎+关键词，零数据源维护成本
3. **不人工审核**：用质量评分的置信度阈值替代人工判断
4. **不追求实时**：日报的 value 是"精选"不是"快"，放弃 Twitter Stream 等伪需求
5. **以自动优化为目标**：日报是自动优化引擎的第一个真实用例，每次生成都让系统更聪明

领主将军阁下，如无异议，可按此规划推进 Phase 1（日报任务类型）的文档设计。
